Stop Re-Remembering the Basics

A fixed template so you never web-search "which header is iota in" again — plus the 20-second ritual that kills your edge-case misses.

Why this lesson, now. On CSES 1083 you lost time researching how to fill a vector and which header iota lives in, then you missed an overflow edge case on 1068 that you already knew about. Neither is a knowledge gap. Both are friction — and friction is fixable with two habits: one template, one checklist.

1. The template (paste it, every time)

The single line #include <bits/stdc++.h> pulls in every standard header — <vector>, <numeric> (that's where iota hid), <algorithm>, <set>, all of it. So the "which header?" question disappears permanently.

#include <bits/stdc++.h>
using namespace std;
using ll = long long;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    // solve here

    return 0;
}

Save this as a snippet or keep it in a template.cpp you copy. The using ll = long long; line is your overflow insurance — when a constraint tells you the sum won't fit int, you just type ll. Full syntax lives in the STL cheat sheet — print it.

2. The five STL tools that carry easy problems

Most Div-2 A/B and regional-easy problems need only these. You'll recognize every one — this is reactivation, not new material:

  • vector<T> a(n) — your default array. Read with for (auto& x : a) cin >> x;
  • sort(a.begin(), a.end()) — half of all easy problems start here.
  • iota(a.begin(), a.end(), 0) — fill 0,1,2,… (the one you searched for).
  • accumulate(a.begin(), a.end(), 0LL) — sum. The 0LL seed prevents the exact overflow you hit.
  • set / map — "have I seen this?" and "how many of each?" — s.count(x), cnt[x]++.

3. The pre-submit checklist — your actual weak point

You said it yourself: "I still missed an edge case." Strong solvers don't have better memory — they run a fixed 20-second scan before every submit. Burn this in until it's automatic:

#Before you submit, ask…
1Overflow? Any sum/product > 2·10⁹? → ll. (this is the one that keeps getting you)
2Smallest input? n = 1, n = 0, empty — does it still work?
3Off-by-one? < vs <=, 0-based indices.
4Reset per test case? Every array/counter cleared inside the t loop?
5Read all input? Consumed every number the format promised?
