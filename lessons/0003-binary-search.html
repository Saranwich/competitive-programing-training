Binary Search on the Answer

The technique that turns "try every possible answer" (too slow) into "test 60 answers" (instant) — and the single biggest points jump from your level.

Why this matters most. Searching a sorted array you can already do with lower_bound. The real skill — the one that unlocks a whole tier of regional problems — is realizing that the answer itself can be binary-searched, even when there's no array in sight. Master this and problems that looked impossible become 15-line solutions.

The core idea: monotonic yes/no

Binary search needs one thing: a test check(x) whose answer goes F F F T T T as x increases and never flips back. That's monotonicity. When you have it, you don't try all values — you halve the range each step, so 10¹⁸ possible answers cost only ~60 tests.

The tell in a problem statement: "find the minimum X such that…" or "the maximum X such that…" where making X bigger only ever makes the situation easier (or harder), never both.

Worked example — Factory Machines (CSES 1620)

n machines, machine i makes one product every k[i] seconds. Minimum time to make t products total?

Trying every time value up to 10¹⁸ is impossible. But ask the question backwards:

"Given x seconds, can we make at least t products?" → In x seconds machine i makes x / k[i] products. Sum them. Is the sum ≥ t?

That test is monotonic: more time never makes fewer products. So F F F T T T — binary search the smallest x that passes:

bool check(ll x, vector<ll>& k, ll t) {
    ll made = 0;
    for (ll ki : k) {
        made += x / ki;
        if (made >= t) return true;   // cap to avoid overflow
    }
    return made >= t;
}
// ... in main, with k read and t given:
ll lo = 1, hi = 2e18;
while (lo < lo + (hi - lo)) {           // i.e. while (lo < hi)
    ll mid = lo + (hi - lo) / 2;
    if (check(mid, k, t)) hi = mid;
    else                  lo = mid + 1;
}
cout << lo;                              // smallest time that works

The whole solution is the template from the binary search cheat sheet plus a 4-line check. That's the pattern for an entire family of problems.

The 3 traps (all on the cheat sheet)

  • Overflow in the midpoint: (lo+hi)/2 overflows when both are ~10¹⁸. Always lo + (hi-lo)/2.
  • hi too small: if your upper bound can't reach the true answer, you'll return garbage. Pick a provably-safe hi (here 2·10¹⁸ > any possible time).
  • check overflows: summing x/k[i] across n machines can blow past ll — so return true the moment you reach t (the if (made >= t) cap above).
