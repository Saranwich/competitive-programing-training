Two Pointers & Prefix Sums

Two cheap tricks that turn an O(n²) subarray scan into O(n). Together they cover a huge slice of easy–medium regional problems — and they pair naturally, so you learn both today.

Same enemy, two weapons. Both attack the same monster: "look at every subarray/pair", which is O(n²) and dies at n = 2·10⁵. Prefix sums precompute so any range sum is O(1). Two pointers slide a window so each element is visited a constant number of times. Reach for one the moment you see "subarray", "contiguous", "pair", or "window".

Part 1 — Prefix sums

Build one array once: p[i] = sum of the first i elements. Then the sum of any range [l, r] (0-indexed, inclusive) is one subtraction:

vector<ll> p(n + 1, 0);
for (int i = 0; i < n; i++) p[i + 1] = p[i] + a[i];

// sum of a[l..r] inclusive:
ll s = p[r + 1] - p[l];

The n + 1 size and the +1 offset are the whole trick: p[0] = 0 means "sum of nothing", so l = 0 needs no special case. Always make p a long long — a sum of 2·10⁵ values up to 10⁹ overflows int instantly.

The move that wins problems: prefix + hashmap

"How many subarrays have sum exactly k?" A range [l, r] has sum k exactly when p[r+1] - p[l] = k, i.e. p[l] = p[r+1] - k. So sweep left to right and, at each prefix, ask "how many earlier prefixes equal (current − k)?" Count them with a map.

map<ll,ll> seen;      // prefix value -> how many times it occurred
seen[0] = 1;          // the empty prefix, so subarrays starting at index 0 count
ll pre = 0, ans = 0;
for (int i = 0; i < n; i++) {
    pre += a[i];
    ans += seen[pre - k];   // earlier prefixes that close a sum-k subarray here
    seen[pre]++;
}

That's O(n log n), and it's the skeleton for a whole family (sum = k, sum divisible by k, etc.). Use map, not unordered_map — remember the anti-hash TLE from Lesson 3's CSES 1621.

Part 2 — Two pointers (sliding window)

Keep a window [l, r]. Push r forward to grow it; when the window breaks a rule, push l forward to shrink it. Both pointers only ever move right, so the total work is O(n) even though it looks like a nested loop.

ll l = 0, sum = 0, ans = 0;
for (ll r = 0; r < n; r++) {
    sum += a[r];                 // grow window to include a[r]
    while (sum > target) {       // shrink until valid again
        sum -= a[l];
        l++;
    }
    if (sum == target) ans++;    // window [l, r] satisfies the rule
}

This exact shape solves "count subarrays with sum = x" for positive numbers (CSES 1660). The key that makes it legal: growing r only raises sum, shrinking l only lowers it — a monotonic knob, same instinct as binary search's check.

Which one? Values can be negative, or you need "sum divisible by k" → prefix + map. Values are positive and you're finding a longest/shortest/counting window → two pointers. When unsure, prefix + map is the more general hammer.
