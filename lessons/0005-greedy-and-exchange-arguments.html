Greedy & Exchange Arguments

Greedy is the cheapest technique to write and the easiest to get wrong. The whole skill is two moves: find the right order, then convince yourself the order is safe. Get those and a big slice of the regional-easy set falls in ten lines of code.

The one-sentence definition. A greedy algorithm builds the answer by repeatedly taking the choice that looks best right now, never reconsidering. It's O(n log n) at worst (usually just a sort plus a linear pass). The catch: locally best is not always globally best — so greedy is the only technique where you must justify it before you trust it.

The trap, first

Make change for 6 using coins {1, 3, 4}. Greedy grabs the biggest coin ≤ remainder: 4, then 1, then 1 — three coins. But 3 + 3 is two. Greedy lost. Same problem, coins {1, 5, 10, 25} — greedy is optimal. The technique isn't wrong or right in the abstract; it depends on the exact structure, and the only way to know is to argue it.

The trust test: the exchange argument

Before you submit a greedy, run this thought in your head:

Take any optimal solution. Show you can swap one of its choices for greedy's choice without making it worse. Repeat, and the optimal solution turns into greedy's — so greedy is at least as good.

You rarely write this down in a contest. But doing it once, informally, is what separates "I guessed greedy" from "I know greedy works here." If you can't find the swap, that's your signal the greedy is wrong — go back to sorting differently, or reach for DP (Lesson 6).

Pattern A — Interval scheduling (the workhorse)

"Given intervals, pick the most that don't overlap." (Movies, meetings, tasks with start/end.) The winning order is sort by ending time, earliest first — then walk left to right, taking an interval whenever it starts at or after the last one you took ended.

sort(v.begin(), v.end(), [](auto& a, auto& b){
    return a.second < b.second;      // by END time
});
int count = 0; ll lastEnd = -INF;
for (auto& [start, end] : v) {
    if (start >= lastEnd) {           // fits after the last one taken
        count++;
        lastEnd = end;
    }
}

Why end time, not start time or length? The exchange argument: the interval that ends soonest leaves the most room for everything after it. Any optimal schedule's first interval can be swapped for the earliest-ending one without losing count — so greedy matches optimal. (Sort by start and one giant early interval blocks everything; sort by length and you can still make a bad first pick.)

Pattern B — Event sweep (count overlaps / allocate resources)

"What's the maximum number of customers in the restaurant at once?" Don't think in intervals — think in events on a timeline. Split each interval into a +1 at its arrival and a -1 at its departure, sort all events by time, and sweep a running counter. The peak of that counter is your answer.

vector<pair<ll,int>> ev;             // (time, +1 arrival / -1 departure)
for (auto& [a, b] : v) { ev.push_back({a, +1}); ev.push_back({b, -1}); }
sort(ev.begin(), ev.end());          // ties: -1 before +1 sorts first, which is correct
int cur = 0, best = 0;
for (auto& [t, d] : ev) { cur += d; best = max(best, cur); }

This sweep idea scales far past today — it's the same skeleton for allocating rooms (add a priority queue of end times) and for line-sweep geometry. Learn the shape now.

Recognising the sort key

90% of greedy is choosing what to sort by. A rough field guide:

The problem asks…Sort by
most non-overlapping intervalsending time, ascending
minimise sum of finish times / penaltiesduration (processing time), ascending
peak simultaneous overlapsplit into ±1 events, by time
match/pair items under a budgetvalue ascending, then two pointers

When a greedy order isn't obvious, try the two or three plausible keys on a tiny example where you know the true answer. If a key fails even one small case, it's dead — no need to prove anything.
