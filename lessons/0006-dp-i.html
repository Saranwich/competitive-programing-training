DP I — State, Transition, Order

When greedy's swap fails, don't give up on trying every choice — just stop recomputing the same subproblem. That's the whole idea of dynamic programming. Nail the 1D version and coin-change, dice, and grid-path problems become fill-in-a-loop.

The one-sentence definition. DP = brute force + memory. You define a subproblem, solve the small ones first, and reuse their answers to build bigger ones — so each subproblem is solved once instead of exponentially many times. If greedy is "one clever choice," DP is "try every choice, but never twice."

Why now: greedy's fallback

Last lesson: coins {1,3,4} making 6 — greedy grabs 4+1+1 (three) but 3+3 is two. No safe swap exists, so greedy is wrong. DP is the answer: to make 6, try every last coin (a 1, a 3, or a 4) and take the best of the smaller subproblems. That "try every last move, reuse subanswers" is the reflex to build today.

The recipe — five questions, every time

Every 1D DP is the same five-line ritual. Answer these in words before you write code:

#QuestionFor "min coins to make x"
1State — what does dp[i] mean?dp[x] = fewest coins summing to exactly x
2Transition — try every last movedp[x] = min over coins c of dp[x-c] + 1
3Base case — smallest subproblemdp[0] = 0 (zero coins make 0)
4Order — fill small → largex from 1 up to target
5Answer — where does it live?dp[target]

Miss any one and the code is wrong. The state definition (Q1) is 80% of the work — get it in one clear sentence and the transition usually writes itself.

Worked example A — counting (sum the transitions)

CSES 1633 Dice Combinations: how many ordered ways to roll a sum of n with dice showing 1–6? The last roll was some value 1..6, so sum the ways to reach what came before it:

const int MOD = 1e9 + 7;
vector<long long> dp(n + 1, 0);
dp[0] = 1;                                 // one way to make 0: roll nothing
for (int x = 1; x <= n; x++)
    for (int d = 1; d <= 6; d++)
        if (x - d >= 0)
            dp[x] = (dp[x] + dp[x - d]) % MOD;   // sum over last roll
cout << dp[n] << "\n";

Counting DP sums; the answer explodes → take it mod 1e9+7 as you go. This is your overflow instinct (LR-0002) in DP form: any "count the number of ways" problem will say "modulo 1e9+7" — apply it inside the loop, never at the end.

Worked example B — optimisation (min/max the transitions)

CSES 1634 Minimizing Coins: fewest coins from set c[] to make x. Same shape, but min instead of sum — and now you need a "impossible" sentinel:

const long long INF = 1e18;
vector<long long> dp(x + 1, INF);
dp[0] = 0;
for (int s = 1; s <= x; s++)
    for (long long coin : c)
        if (s - coin >= 0 && dp[s - coin] + 1 < dp[s])
            dp[s] = dp[s - coin] + 1;
cout << (dp[x] == INF ? -1 : dp[x]) << "\n";   // INF means unreachable
Two bugs to pre-empt (both are your recurring family). (1) The INF guard: only relax from dp[s-coin] if it's reachable — INF + 1 silently becomes a "valid" answer if you skip the check. (2) Bounds: s - coin >= 0 before indexing, or you read negative indices. Same discipline as the seen[0] / >= l guards from Lesson 4.

The one twist: does order matter?

Two problems look identical but aren't. Coin Combinations I counts ordered sequences (2+3 and 3+2 are different) — loop sum outside, coins inside. Coin Combinations II counts sets (2+3 = 3+2) — loop coins outside, sum inside. Same recurrence, swapped loops. When a counting answer comes out too big, this loop order is the first thing to check.

Do it now (~3 hours)

At the top of each file write the five recipe answers as a comment before coding — state, transition, base, order, answer. That comment is the solve; the code is transcription. Run the pre-submit checklist every time.

  1. Counting (sum): CSES 1633 Dice Combinations — worked above, type it from your own recipe, not the lesson. ~15 min.
  2. Optimisation (min + INF): CSES 1634 Minimizing Coins — mind the sentinel and the -1 output. ~20 min.
  3. Ordered counting: CSES 1635 Coin Combinations I — sum outside, coins inside, mod as you go. ~20 min.
  4. Unordered counting (loop-order twist): CSES 1636 Coin Combinations II — flip the loops. Prove to yourself why. ~30 min.
  5. Different transition: CSES 1637 Removing Digits — state dp[x] = min steps to 0; the "moves" are the digits of x. ~25 min.
  6. Stretch (reconstruct the answer): CSES 1746 Array Description or come back to it — DP where you also print which choices. ~40 min.

Get 1–5; #6 is the reach. For every one, say your state definition out loud in one sentence before coding — if you can't, that's the signal to ping me, not to start typing.
