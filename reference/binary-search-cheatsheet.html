Binary Search

Two jobs: find a value in a sorted array (use the STL), and find the best answer in a monotonic search space (write the template).

The one idea. Binary search works whenever the answer space is monotonic: a yes/no test that is F F F T T T as you increase x. Each step halves the range → O(log n). If you can phrase "is x good?" and the goodness never flips back, you can binary search it.

These are O(log n) and already correct. Hand-rolling this loop is where off-by-one bugs live — let the STL do it.

Job 2 — binary search on the answer (the template)

When the problem asks for a minimum/maximum value that works and "works" is monotonic. Memorize this shape:

// smallest x in [lo, hi] with check(x) == true   (F F F T T T)
ll lo = 0, hi = 2e18;          // bounds MUST cover the true answer
while (lo < hi) {
    ll mid = lo + (hi - lo) / 2;   // avoids overflow
    if (check(mid)) hi = mid;      // mid works -> answer is mid or lower
    else            lo = mid + 1;  // mid fails -> answer is strictly higher
}
// lo == hi == the answer

For the mirror problem (largest x that works, T T T F F F): swap to mid = lo + (hi - lo + 1)/2, if (check(mid)) lo = mid; else hi = mid - 1;

Writing check(x). Assume you're given x for free, then answer "is x enough?" — usually one O(n) loop. Example: "can we finish in x time?" → sum up how much each machine/worker produces in x time, compare to target.

Three traps

TrapFix
(lo+hi)/2 overflowsuse lo + (hi-lo)/2
hi too small to reach answerset hi to a provably-safe upper bound (often 2·10¹⁸)
check itself overflowscap the running total, or use ll / __int128

Is it monotonic? (the go/no-go check)

Before coding, sanity-check: "if x works, does every larger x also work?" If yes → binary search. If the goodness can turn off again as x grows, it's not monotonic and BS won't apply.
