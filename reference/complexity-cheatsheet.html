Constraints → Complexity

The first thing to compute on every problem, before any algorithm.

The 10⁸ rule. A judge runs roughly 10⁷–10⁸ simple operations per second. With a 1 s limit, plug the maximum n into your algorithm's complexity — the result must stay under ~10⁸.
Max input sizeAllowed complexityTypical technique
n ≤ 10O(n!)try all permutations
n ≤ 20O(2ⁿ)try all subsets, bitmask
n ≤ 500O(n³)triple loop, Floyd–Warshall
n ≤ 5 000O(n²)double loop, simple DP
n ≤ 10⁶O(n log n)sorting, binary search, heaps
n ≤ 10⁸O(n)single pass, two pointers, prefix sums
n > 10⁸ (or 10¹⁸)O(log n) / O(1)binary search on answer, math formula

Table adapted from Competitive Programmer's Handbook §2.3 (resources/books/book.pdf, p. 21).

Traps

  • Constant factors: heavy operations (modulo, map lookups, string copies) can make an O(n log n) behave like 5–10× slower. If it's borderline, prefer arrays over map.
  • Multiple test cases: if t ≤ 10⁴ and n ≤ 10⁵ but "sum of n over all test cases ≤ 2·10⁵", budget against the sum, not t·n.
  • Overflow: n ≤ 10⁵ values ≤ 10⁹ → sums reach 10¹⁴. Use long long.
