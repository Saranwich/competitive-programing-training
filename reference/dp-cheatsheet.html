DP I — 1D Dynamic Programming

Brute force + memory. Define a subproblem, solve small ones first, reuse them. Each state solved once.

The five-question recipe. Answer in words before writing code: 1. State — what is dp[i]? · 2. Transition — try every last move · 3. Base case — smallest subproblem · 4. Order — fill small→large · 5. Answer — where does it live? State (Q1) is 80% of the work.

Counting DP — SUM the transitions

const int MOD = 1e9 + 7;
vector<long long> dp(n + 1, 0);
dp[0] = 1;                              // one way to make 0
for (int x = 1; x <= n; x++)
    for (int move : moves)
        if (x - move >= 0)
            dp[x] = (dp[x] + dp[x - move]) % MOD;

"Number of ways" ⇒ sum, and it overflows ⇒ mod 1e9+7 inside the loop. Base case is dp[0] = 1.

Optimisation DP — MIN / MAX the transitions

const long long INF = 1e18;
vector<long long> dp(x + 1, INF);
dp[0] = 0;
for (int s = 1; s <= x; s++)
    for (long long c : coins)
        if (s - c >= 0 && dp[s - c] + 1 < dp[s])
            dp[s] = dp[s - c] + 1;
cout << (dp[x] == INF ? -1 : dp[x]);    // INF = unreachable

Base case is dp[0] = 0. Answer is dp[target].

Recognising a DP problem

  • "Number of ways to…" / "minimum/maximum … to reach …"
  • Greedy's exchange swap fails — choices interact.
  • Small target/state space (n ≤ ~1e6 for 1D, product ≤ ~1e7 for 2D).
  • You can phrase "the last move was ___" and the rest is a smaller identical problem.
Contest move. Write the five recipe answers as a comment first. If you can't state dp[i] in one sentence, you don't have a DP yet — go back to the state, not the code.
