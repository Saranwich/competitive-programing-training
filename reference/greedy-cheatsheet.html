Greedy & Exchange Arguments

Take the locally best choice, never look back. O(n log n) — usually a sort plus one pass. The catch: prove it's safe before you trust it.

The trust test. Take any optimal solution; show you can swap one of its choices for greedy's without making it worse. If you can find that swap, greedy is optimal. If you can't, greedy is wrong → sort differently, or use DP.

Pattern A — Interval scheduling (max non-overlapping)

sort(v.begin(), v.end(), [](auto& a, auto& b){
    return a.second < b.second;      // by END time, ascending
});
int count = 0; ll lastEnd = -INF;
for (auto& [start, end] : v)
    if (start >= lastEnd) { count++; lastEnd = end; }

Sort by end — earliest-ending leaves the most room. NOT start, NOT length.

Pattern B — Event sweep (peak overlap / resources)

vector<pair<ll,int>> ev;             // (time, +1 arrival / -1 departure)
for (auto& [a, b] : v) { ev.push_back({a,+1}); ev.push_back({b,-1}); }
sort(ev.begin(), ev.end());          // ties: -1 sorts before +1, which is correct
int cur = 0, best = 0;
for (auto& [t, d] : ev) { cur += d; best = max(best, cur); }

Min rooms to fit all events = same sweep, add a priority_queue of end times (free a room when its end has passed).

When it's NOT greedy

  • Coin change, arbitrary coins — {1,3,4} for 6: greedy 4+1+1 loses to 3+3. Use DP.
  • Choices interact / future depends on the whole prefix — no local swap works → DP.
  • Can't name the sort key in one sentence — probably not greedy.
Contest move. Can't prove the exchange? Test 2–3 candidate keys on a tiny hand-checked case. A key that fails one small case is dead instantly — no proof needed. Only argue the survivor.
